02 — Incongruencias críticas

Estas contradicciones entre los documentos y el código pueden causar problemas reales si no se resuelven antes de seguir construyendo.

⚠️
Stack duplicado: Flutter vs React Native
El PDF de arquitectura especifica React Native + Expo con carpetas mobile/app/(auth)/login.tsx, hooks en TypeScript, etc.
El código entregado es Flutter (Dart) puro. Son stacks completamente distintos — no se comparte nada.

Impacto: Toda la arquitectura de carpetas del PDF es inválida para el código real. El README incluso dice "Built with Rocket.new" y describe una app genérica Flutter, no CycleCoach.
⚠️
Backend inexistente en el repo
Los PDFs documentan un backend Node.js completo con 15+ endpoints, Prisma, cron jobs, AIRouter...
El repo sólo contiene la app Flutter. No hay carpeta backend/. Todo el "código" del backend está solo en PDFs como texto, sin un proyecto real.

Impacto: El prototipo es solo frontend con mocks. No hay integración real posible sin crear el backend desde cero.
⚠️
env.json con 7 claves vs .env del PDF con 10
El env.json incluye PERPLEXITY_API_KEY y GOOGLE_WEB_CLIENT_ID que no aparecen en ningún documento.
El PDF de pagos define STRIPE_PRICE_ID_MONTHLY, STRIPE_PRICE_ID_ANNUAL, REDIS_URL, NODE_ENV que no están en el env.json.

Impacto: Ambos son incompletos y ninguno es fuente de verdad.
⚠️
Modelo Claude incorrecto en el código
El PDF de pagos usa correctamente claude-haiku-4-5-20251001 y claude-sonnet-4-20250514.
El PDF de arquitectura usa claude-sonnet-4-20250514 ✓ para aiClient pero en el AIRouter define el provider como string 'claude-sonnet' y 'claude-haiku' sin el model string correcto en el switch-case.

Impacto: El AIRouter fallará en producción si se copia tal cual — el switch mapea 'claude-haiku' pero no usa el model ID correcto claude-haiku-4-5-20251001.
⚠️
AppTheme.lightTheme = darkTheme (siempre oscuro)
En app_theme.dart: static ThemeData get lightTheme => darkTheme;
Y en main.dart: themeMode: ThemeMode.light → la app siempre aplica el tema oscuro aunque el dispositivo esté en modo claro. Es un bug intencional-accidental.

Impacto: Correcto para la estética oscura del producto, pero engañoso. Debería ser ThemeMode.dark directamente o eliminar lightTheme.
⚠️
README totalmente genérico — no describe CycleCoach
El README habla de "flutter_template", "splash_screen" y "Built with Rocket.new". No menciona ni una sola feature de CycleCoach.

Impacto: Si alguien nuevo entra al repo, no sabe qué es el proyecto. Problema de onboarding y presentación a inversores/equipo.
⚠️
pubspec.yaml incluye dependencias no usadas
universal_html, web, fl_chart — ninguna aparece en ningún widget del código. connectivity_plus tampoco está integrada.

Impacto: Binarios más grandes, posibles conflictos de versiones, confusión sobre qué se usa realmente.

03 — Lo que va bien ✅

✅
AppTheme es sólido y consistente
El sistema de tokens (colores de fase, espaciado, radios, duraciones) está bien pensado. Los 4 colores de fase son distintos e identificables. El uso de withAlpha() en lugar de withOpacity() es la práctica correcta en Flutter 3.x.
✅
AIRouter y sistema de fallback — excelente diseño
La lógica de 3 niveles (premium/standard/economy) con detección de keywords de crisis es elegante y directamente productizable. El fallback a respuestas estáticas por fase garantiza UX degradada de calidad. El caché Redis por fase es el correcto punto de ahorro.
✅
Webhook Stripe — implementación correcta
El manejo de express.raw() para verificar firma, responder siempre 200, y manejar los 4 eventos clave es textbook. El patrón es production-ready tal como está.
✅
Detección de crisis en el chat
El array _crisisKeywords en Flutter y la lógica detectTier() en el AIRouter son consistentes conceptualmente (aunque son stacks distintos). La UX de "modo crisis" con banner y cambio de respuesta es el diferencial correcto.
✅
Animaciones de entrada en CyclePhaseScreen
El sistema de _sectionFades + _sectionSlides con intervals escalonados es la forma correcta de hacer entrance animations en Flutter. Bien implementado y sin sobrecargar el vsync.
✅
Modelo de datos Map-first con clases auxiliares
El patrón _partnerProfileMap → _PartnerProfile.fromMap() es correcto para un prototipo: fácil de reemplazar por respuesta de API sin refactorizar la UI.
✅
Prompt engineering bien estructurado
El buildSystemPrompt() combina fase + personalidad + estilo de conflicto + lenguaje de amor en un prompt cohesivo. Las reglas de respuesta (directe, 3-4 puntos, en español) son precisas. Es superior a la mayoría de prompts de apps similares.
✅
Estimación de costos realista
Los cálculos de ~750 tokens/interacción y el ahorro del 82% con el AIRouter son defendibles y honestos. El escenario de 500 usuarios a $1.500/mes con $83 de costo es conservador y creíble para pitches.

04 — Flutter: Issues y mejoras

🔴
Sin state management real — TODO pendiente en producción
Todos los pantallas tienen comentario // TODO: Replace with Riverpod for production. El estado se maneja con setState local, lo que va a crear problemas de sincronización cuando el chat y el dashboard necesiten compartir el profile.
→ Instalar riverpod + flutter_hooks. Crear ProfileProvider, ChatProvider. Es el paso 1 antes de conectar la API real.
🔴
Navegación al Chat rompe el estado del bottom nav
El tab "Chat" hace Navigator.pushNamed() en lugar de cambiar el body. Resultado: cuando el usuario vuelve del chat, el _selectedTab sigue en 0 (Inicio) aunque visualmente esté en Inicio. El tab Chat nunca se activa visualmente.
→ Usar IndexedStack con 4 pantallas o PageView, y manejar el chat como pantalla modal/sheet en lugar de route independiente.
🟡
TypingIndicator hardcodea el color de OVULATION
El indicador de escritura usa AppTheme.phaseOvulationLight y AppTheme.phaseOvulation directamente, ignorando la fase actual del usuario. Si la pareja está en fase menstrual, el indicador será dorado en lugar de rojo.
→ Pasar phase al TypingIndicator como parámetro y usar AppTheme.phaseColor(phase).
🟡
AiChatScreen tiene el phase hardcoded como constante
static const String _currentPhase = 'OVULATION' y static const int _dayOfCycle = 14. La AppBar muestra "Día 14" siempre. El PhaseContextStrip también recibe estos valores fijos.
→ Pasar los datos del profile como argumento de ruta: Navigator.pushNamed(context, AppRoutes.aiChatScreen, arguments: profile) y leerlos en initState.
🟡
_generateMockResponse usa _dayOfCycle que siempre es 14
La respuesta mock menciona "día $_dayOfCycle" en el texto — siempre dirá "día 14" independientemente del ciclo real.
→ Corregir junto con el punto anterior. Prioridad baja hasta conectar la API real.
🟡
Sidebar del chat duplica datos hardcoded
El tablet sidebar en AiChatScreen tiene strings literales "Ovulación", "Mixta", "Directa", "Tiempo de calidad" que no vienen de ningún modelo de datos — son constantes hardcoded que no se actualizarán cuando el perfil real cambie.
→ Extraer la lógica de descripción de fase/personalidad a un helper (ya existe en el PDF como PHASE_DESCRIPTIONS etc.) y usarlo aquí.
🟢
Oportunidad: falta pantalla de Onboarding en el initial route
La ruta inicial lleva directamente a CyclePhaseScreen con un mock profile. En producción debe detectar si el usuario tiene perfil configurado: sin perfil → onboarding, con perfil → dashboard.
→ Crear un SplashScreen que lee SharedPreferences. Si no hay profile → /onboarding. Si hay → /cycle-phase.
🟢
Oportunidad: el servicio HTTP (api.ts del PDF) no existe en Flutter
En el prototipo no hay ningún servicio de red. dio está en pubspec pero sin usar. Crear este servicio es el siguiente paso crítico para conectar el backend.
→ Crear lib/services/api_service.dart con Dio, interceptors JWT, y los métodos que mapean a los endpoints del backend.

05 — Backend & IA: Issues y mejoras

🔴
AIRouter: model strings incorrectos en el switch-case
El switch usa 'claude-sonnet' como provider ID, pero dentro del case usa el string correcto 'claude-sonnet-4-20250514'. El problema está en que getStaticFallback() busca systemPrompt.includes('MENSTRUAL') pero el prompt real usa la variable que se construye con PHASE_DESCRIPTIONS[ctx.phase], no el string literal "MENSTRUAL".
→ Pasar la phase como parámetro separado al fallback, no intentar parsear el systemPrompt.
🔴
dailyPhaseCheck.job.ts: llamada a IA sin rate limiting ni concurrencia
El cron itera for (const profile of profiles) de forma secuencial — con 500 usuarios en la misma fase, hace 500 llamadas a la IA en serie durante ~5 minutos. El caché Redis está definido pero el cron llama getAIRecommendation en lugar de routeToAI, saltándose el caché.
→ 1) Usar routeToAI en el cron para aprovechar el caché. 2) Agrupar por fase: generar 1 respuesta por fase y distribuir a todos los usuarios de esa fase.
// Fix propuesto para el cron:
const phaseGroups = groupBy(profiles, p => p.currentPhase);

for (const [phase, users] of Object.entries(phaseGroups)) {
  // 1 llamada a IA por fase (no por usuario)
  const recommendation = await routeToAI(
    buildSystemPrompt({ phase, ...genericCtx }),
    [{ role: 'user', content: 'Dame el consejo del día.' }],
    'economy'
  );
  // Luego enviar push a todos los users de esa fase
  await Promise.all(users.map(u => sendPushNotification(u.userId, {...})));
}
🔴
Falta middleware de suscripción (mencionado en el PDF pero no implementado)
El PDF dice "Middleware de suscripción en endpoints premium" en semana 8, pero no hay ningún código de este middleware. Los endpoints /api/ai/chat y /api/ai/recommend son el core del producto de pago — sin el middleware, cualquier usuario FREE puede acceder.
→ Crear requirePremium middleware que verifique user.plan === 'PREMIUM' y responda 403 con JSON descriptivo si es FREE. Aplicar a las rutas de IA.
// requirePremium.middleware.ts
export const requirePremium = async (req, res, next) => {
  const user = await prisma.user.findUnique({ where: { id: req.userId } });
  if (user?.plan !== 'PREMIUM') {
    return res.status(403).json({ 
      error: 'SUBSCRIPTION_REQUIRED',
      message: 'Esta función requiere plan Premium'
    });
  }
  next();
};
🟡
buildCacheKey puede colisionar con facilidad
La cache key usa las primeras 100 chars del systemPrompt + 50 chars del último mensaje. Dos usuarios en la misma fase con diferente perfil de personalidad podrían generar la misma key si la personalidad está después del carácter 100 del prompt.
→ Incluir la fase explícita y un hash del perfil de personalidad en la key: ai:${phase}:${personalityHash}:${userInputHash}
🟡
handlePaymentFailed no actualiza el plan a GRACE_PERIOD
Cuando falla el pago, se envía push notification pero no se cambia el estado del usuario. Stripe reintenta 3 veces, pero si los 3 fallan y se dispara customer.subscription.deleted, sí se limpia el plan. El problema es el período entre fallo y cancelación: el usuario sigue con acceso premium durante potencialmente 7-14 días sin pagar.
→ En handlePaymentFailed, establecer un campo paymentFailedAt y en el middleware verificar si han pasado más de 3 días desde el fallo.
🟢
Mejora: añadir streaming a los endpoints de chat
Actualmente el chat espera la respuesta completa (~1.5-2s). Con streaming, el usuario ve texto aparecer en tiempo real desde los primeros 300ms. Anthropic soporta SSE nativamente.
→ Usar client.messages.stream() de Anthropic SDK y enviar chunks via SSE. En Flutter, usar dio con responseType: ResponseType.stream.

Nota importante: Push Notifications en Flutter

El PDF usa Expo Notifications porque asume React Native + Expo. Con Flutter, la solución correcta es firebase_messaging (Flutter plugin oficial de Firebase/FCM) + APNs para iOS. El cron del backend debería usar el Admin SDK de Firebase para enviar pushes directamente a los tokens FCM, no Expo push tokens.

09 — Resumen ejecutivo

🎯

El concepto es sólido

El diferencial (perfil de personalidad + ciclo = consejo contextual) es genuinamente diferente a las apps de ciclo existentes. El prompt engineering está por encima de la media.

⚠️

La mayor deuda: documentos vs código

Los PDFs documentan un stack React Native que no existe. El código real es Flutter. Esta brecha es la raíz de la mayoría de incongruencias y debe resolverse primero.

🏗️

El backend es el gap más grande

Existe bien diseñado en papel, pero no como proyecto. El AIRouter, el webhook de Stripe y el sistema de caché son excelentes — pero están atrapados en PDFs.

🚀

Próximo milestone realista

MVP funcional en 3-4 semanas: backend inicializado + onboarding conectado + chat con IA real (sin streaming) + acceso libre (sin paywall). Suficiente para validar la propuesta de valor.
